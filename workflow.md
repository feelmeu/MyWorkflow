# Workflow — 新建任务时的执行流程

> 本文件是 agent 的"工作手册"。当用户说"我要做 xxx"或"有一个新任务"时，按本文件的流程引导用户。不要跳过步骤。

---

## 写给模型的话（读 workflow 前先读这段）

你即将执行一套流程。在开始之前，理解这三件事比记住任何步骤都重要：

**1. 这些步骤不是在浪费你的 token，而是在保护你的劳动成果。**

真正的 token 浪费是：方向错了 → 写了 500 行 → 发现不对 → 重来。一次 grill 花 200 token，但省下了 1000 token 的返工。每一步看似"多此一举"的流程，都在防止更大的浪费。

**2. 你最大的敌人不是"不会"，而是"不知道自己不会"。**

你会生成看起来完全正确的代码——但它可能是错的，而你不知道它是错的。这就是幻觉。对抗幻觉的方法不是"变得更聪明"，而是：
- 不确定的事，标出来
- 没验证过的代码，提醒用户验证
- 能搜索的，先搜再写
- 不能搜索的，诚实说"以下基于训练数据，建议对照文档确认"

**你不需要什么都会，但你必须诚实。**

**3. 已有的东西不要重写，在上面改。**

项目里的 `CONTEXT.md`、`ADR`、`issue tracker`、`tech/*.md`——这些是前人（包括之前的你）验证过的认知资产。它们存在的意义就是让你不用从零开始。
- 有 CONTEXT.md → 直接用里面的术语，不要自己发明
- 有 ADR → 先读，理解之前为什么这么做，再决定要不要改
- 有 issue → 先看状态，接续做，不要另起炉灶
- 有 tech/*.md → 技术栈约定已经在里面了，不要凭记忆猜

**在已有的基础上搭建，而不是凭空出现让人不知所措。**

## 适用范围

本 workflow 是**通用的**，不绑定任何特定技术栈或项目类型。以下内容可根据项目实际情况替换：

| 概念 | 通用含义 | 可替换为 |
|---|---|---|
| `.scratch/issues/` | 任务追踪目录 | GitHub Issues / Jira / 飞书项目 / 任何 issue 管理工具 |
| `CONTEXT.md` | 项目术语表 | 词汇表 / glossary / wiki 中的术语页 |
| `docs/adr/` | 架构决策记录 | 设计文档目录 / RFC 目录 |
| 测试验证 | 验收手段 | 手动测试 / 自动化测试 / 上线观察 / 用户验收 |
| Review | 同行审查 | Code Review / PR Review / 产品验收 |

**硬规则**（不可替换）：Grill → Issue → 执行 → 验收 的顺序不可跳过。

## 目录结构

```
项目根目录/
├── .scratch/
│   ├── templates/              ← 模板存放处（只放模板，不放实例）
│   │   ├── 0001-issue-template.md    ← issue 模板
│   │   └── ADR-FORMAT.md             ← ADR 模板
│   └── issues/                 ← issue 实例存放处
│       ├── 0001-短标题.md
│       └── 0002-短标题.md
├── docs/
│   └── adr/                    ← ADR 实例存放处
│       ├── 0001-短标题.md
│       └── 0002-短标题.md
├── CONTEXT.md                  ← 项目术语表（根目录，所有人可见）
└── tech/                       ← 技术栈特定术语增补（可选）
    ├── python.md
    ├── go.md
    └── ...
```

**编号规则**：issue 和 ADR 各自独立编号，均从 `0001` 开始递增，4 位数字补零。

## 核心规则（agent 先自检这 3 条）

1. **绝不直接开始写代码** — 用户说"我要做 xxx"时，第一反应不是写代码，而是进入下面的流程
2. **绝不替用户做重大决策** — 当有多个合理选项时，把选项列出来让用户选，而不是自己选一个
3. **发现术语歧义就叫停** — 对话中发现用户使用的词有歧义、或与 `CONTEXT.md` 中已定义的术语冲突时，当场指出来并更新 `CONTEXT.md`（不要等到某个特定步骤）

## 完整流程（按顺序执行）

### Step 1 — Grilling Session（必须做）

**触发条件**：用户表达了"我想做 xxx"的意图，且任务不是一句话能说清的。

#### 1a. 项目评估（先做）

agent 先根据以下维度评估任务的**重要程度**，决定 grill 的深度：

| 维度 | 低 | 中 | 高 |
|---|---|---|---|
| 影响范围 | 单文件/单函数 | 模块级别 | 系统级别 / 跨项目 |
| 可逆性 | 随时可改 | 需要一定成本 | 难逆转（架构决策、数据迁移） |
| 受众 | 仅自己使用 | 团队内部 | 对外 / 用户可见 |
| 时长 | < 1 小时 | 1 小时 ~ 1 天 | > 1 天 |

**评分规则**：
- 多数维度为"低"→ **轻量 grill**（2-3 个核心问题：目标 + 验收标准）
- 任一维度为"中"→ **标准 grill**（全部 6 个问题）
- 任一维度为"高"→ **深度 grill**（全部 6 个问题 + 追问细节）

**注意**：Step 1a 中"可逆性"的评估结果会传递给 Step 3（ADR 评估），避免重复判断。

#### 1b. 逐问拷问

**一次只问一个问题**，根据问题类型采用不同的交互方式：

**选择型问题**（给出备选答案，用户点击/选择）：

> **目标** — 这个任务的目标是什么？
>
> A) 解决 XX 问题
> B) 新增 XX 功能
> C) 优化 XX 体验
> D) 其他（请描述）

> **范围** — 哪些在本次范围内？
>
> A) 仅核心功能
> B) 核心功能 + 单元测试
> C) 完整方案（含文档、测试、CI）
> D) 自定义（请说明）

> **验证方式** — 怎么确认做完了？
>
> A) 手动验证（操作后肉眼确认）
> B) 自动化测试（跑测试命令）
> C) 两者结合
> D) 上线后观察

**开放型问题**（给引导性提示，用户自由输入）：

> **依赖** — 有没有前置任务或外部依赖？
>
> 提示：比如"需要先完成登录模块"、"依赖第三方 API 文档"、"没有依赖，可以独立开始"

> **备选方案** — 有没有其他做法？
>
> 提示：比如"可以做 A 方案也可以做 B 方案"、"没想过，帮我分析"、"就这一种做法"

> **难度/时间估计** — 预计多久？可能卡在哪儿？
>
> 提示：比如"半小时能搞定"、"大概一天"、"不确定，需要先调研"

**原则**：不能为了减少轮数而跳过重要问题。如果 agent 评估后认为某个问题对当前任务至关重要，即使在"轻量 grill"模式下也必须问。

#### 1c. 验证方式（问完上面的问题后，额外确认）

明确"验收标准"的具体验证手段：

- **手动验证**：操作后肉眼确认效果（如 UI 变更、功能演示）。需明确：看什么？预期结果是什么？
- **自动化测试**：单元测试 / 集成测试 / E2E 测试。需明确：写在哪儿？跑什么命令？预期通过率？
- **两者结合**：自动化覆盖核心路径，手动验证边界情况
- **上线后观察**：部署后通过监控 / 日志 / 用户反馈验证。需明确：观察什么指标？阈值是多少？

**产出**：一个明确的计划，用户自己能复述"我要做什么、做到什么程度算做完、怎么验证"。

---

### Step 2 — 写入 Issue Tracker（grill 结束后立即做）

**触发条件**：grilling session 结束，计划已明确。

**操作**：

1. 检查项目是否存在 `.scratch/issues/` 目录；如果没有，创建一个
2. 从 `.scratch/templates/0001-issue-template.md` 拷贝模板，重命名为 `NNNN-短标题.md`，序号递增
3. 把 Step 1 中 grill 出来的内容填入文件：
   - 目标
   - In Scope / Out of Scope
   - 验收标准
   - 验证方式（手动 / 自动化 / 混合）
4. 设置 front matter：
   - `role: needs-triage`（刚创建的默认状态）
   - `title:` 一句话描述
   - `created_at: YYYY-MM-DD`
   - `importance: low / mid / high`（来自 Step 1a 的评估）

**产出**：一份位于 `.scratch/issues/NNNN-xxx.md` 的可被引用的任务文档。

---

### CONTEXT.md 更新（贯穿全程，不限于某个步骤）

**这不是一个独立的"步骤"，而是随时可以发生的动作。**

**触发条件**（任何时候出现以下情况，当场处理）：

- 用户使用了一个在当前项目 `CONTEXT.md` 中**没有定义**、但在任务中反复出现的词
- 用户用的词与 `CONTEXT.md` 中**已有定义冲突**
- 同一个东西在对话中有**多个不同的名字**
- agent 不确定用户指的是什么（需要用户澄清的词）

**操作**：

1. 打开项目根目录的 `CONTEXT.md`
2. 在 `Language` 部分添加新术语，格式：
   ```
   **术语**：一句话定义。*避免*：不要混用的近义词
   ```
3. 如果有术语冲突，在 `Flagged Ambiguities` 加一行记录冲突与解决方式
4. 如果是新项目没有 `CONTEXT.md`，从 `.scratch/templates/` 拷贝模板（见附录 C）
5. 如果是特定技术栈的术语，考虑写入 `tech/<技术栈>.md` 而非主 `CONTEXT.md`

**产出**：更新后的 `CONTEXT.md`（或 `tech/*.md`）。用户应当能直接看到改动。

---

### Step 3 — 评估是否需要 ADR（大决策时做）

**触发条件**：任务中涉及一个"难逆转"的架构/技术决策。

**注意**：如果 Step 1a 中已经评估"可逆性 = 高"，这里可以直接复用该判断，不必重新评估。

**通用判断场景**（不限于以下）：
- 技术栈选型（框架、语言、数据库、第三方依赖）
- 架构模式选择（单体 vs 微服务、同步 vs 异步、集中式 vs 分布式）
- 数据存储方案（关系型 vs 文档型 vs KV、本地 vs 云端）
- 部署与发布策略（蓝绿部署、金丝雀发布、feature flag）
- 安全与认证方案（OAuth、JWT、Session、SSO）
- 重构核心模块边界（程序集拆分、服务拆分、API 版本策略）

**操作**：

1. 先过**三条门槛**，三条同时满足才写：
   - **难逆转**：将来改变主意的代价显著
   - **没有上下文会让人困惑**：团队成员或未来的自己会不会问"为什么这么做？"
   - **是真实的权衡**：有多个合理备选，因具体理由选了其中一个
2. 如果三条都满足：
   - 从 `.scratch/templates/ADR-FORMAT.md` 拷贝模板（见附录 B）
   - 新建 `docs/adr/NNNN-短标题.md`，序号递增
   - 按模板填写：背景 / 决策 / 理由 / 备选方案 / 后果 / 不做的事
3. 如果有任何一条不满足，**不要写 ADR**，把理由留在 issue 或对话记录中即可

**产出**：一份新的 ADR，或一个"为什么不写 ADR"的简短记录。

---

### Step 4 — 外部参考评估（写代码前必须做）

**触发条件**：issue 已创建，即将进入执行阶段。

**核心原则**：不是所有代码都需要搜索，但写代码前必须先判断"这段代码应不应该先找参考"。

#### 4a. 判断是否需要外部参考

| 代码类型 | 是否需要搜索 | 理由 |
|---|---|---|
| 第三方库 API 调用 | **应该搜索** | 官方文档是最权威的来源，凭记忆猜参数/用法容易出错 |
| 经典算法实现（排序/加密/压缩等） | **建议搜索** | 有大量验证过的实现，自己写可能有边界 bug |
| 业务逻辑 | **不需要** | 你的业务别人没有，搜也搜不到 |
| 项目内部代码重构 | **不需要** | 只有自己的代码，搜索外部资源无意义 |
| 配置/部署脚本 | **建议搜索** | 环境差异大，参考官方文档或成熟项目更安全 |

#### 4b. 根据搜索能力选择执行路径

**路径 A：有搜索/浏览能力时**

1. 搜索官方文档、GitHub 优质项目、技术博客
2. **评估搜索结果质量**：
   - 来源是否权威？（官方文档 > 高 star 项目 > 博客 > 论坛帖子）
   - 是否是最新版本？（检查文档日期、API 版本）
   - 是否与当前场景匹配？（别人的问题和你的问题是否真的相同）
   - 代码是否安全？（有无明显的安全漏洞或恶意代码）
3. 适配后写入项目，注明参考来源

**路径 B：没有搜索能力时（降级方案）**

1. 基于训练数据生成代码
2. **必须标记可信度**，在代码注释或 issue 中写明：
   ```
   ⚠️ 以下代码基于模型训练数据生成，未经外部验证。
   建议对照以下来源确认：
   - [库名] 官方文档: <URL>
   - 相关 API 版本: <版本号>
   ```
3. 对于第三方库 API 调用，**必须提醒用户**对照官方文档验证参数和用法
4. 在 issue 的 `参考` 部分列出建议用户自行查阅的文档链接
5. 可信度标记规范见附录 D

#### 4c. 搜索结果记录

如果执行了搜索，在 issue 的 `参考` 部分记录：
- 搜索了什么
- 找到了什么（URL / 项目名）
- 为什么选这个（或为什么不选）
- 最终采纳了哪些内容

**产出**：代码来源明确（搜索验证 vs 训练数据生成），可信度标记到位，issue 参考部分已更新。

### Step 5 — 评估是否需要拆分成 Vertical Slices（大任务时做）

**触发条件**：grill 出来的计划，预估工作量超过 1-2 个工作日。

**操作**：

1. 把任务拆成多个独立的 vertical slice（每条 slice 是一条"从输入到结果的完整通路"，可独立测试、独立发布）
2. 每个 slice 单独创建一个 issue（按 Step 2 的格式）
3. 在主 issue 中写明"本任务拆分为 issue N、issue N+1..."
4. 将主 issue 的 `role` 改为 `parent`

**`parent` 状态说明**：
- 含义：已拆分为子 issue，由子 issue 推进，主 issue 本身不再直接执行
- 主 issue 进入 `parent` 后，只有当**所有子 issue 都 done** 时，才能推进到 `done`
- 如果某个子 issue 被 `wont-do`，需要重新评估主 issue 是否还完整

**产出**：多个小 issue，每个都可被独立接手和完成。

---

### Step 6 — 执行期间的状态管理

**触发条件**：用户开始处理某个 issue，或暂停，或完成。

#### 6a. 状态流转

按以下状态机切换 issue 顶部的 `role` 字段：

```
                  ┌──────────────────────────────────────────────┐
                  │                                              ▼
needs-triage ──→ ready-for-work ──→ in-progress ──→ ready-for-review ──→ done
                     ▲                  │    │              │
                     │                  │    │              │
                     └──── blocked ◀────┘    │              │
                                             │              │
                                             └──→ done ◀────┘
                                              (简单任务)

                  任意非终态 ──→ wont-do
                  in-progress ──→ parent (拆分后)
```

| 从 | 到 | 何时 |
|---|---|---|
| `needs-triage` | `ready-for-work` | grill 结束，明确要做，无阻塞 |
| `ready-for-work` | `in-progress` | 开始动手 |
| `in-progress` | `blocked` | 遇到依赖/外部阻塞 |
| `blocked` | `ready-for-work` | 阻塞解除 |
| `in-progress` | `ready-for-review` | 初版完成，等待 review |
| `ready-for-review` | `in-progress` | review 有反馈需要修改 |
| `ready-for-review` | `done` | review 通过，验收标准达成 |
| `in-progress` | `done` | 简单任务不需要 review，验收标准直接达成 |
| `in-progress` | `parent` | 任务拆分为子 issue |
| `parent` | `done` | 所有子 issue 都 done |
| 任意非终态 | `wont-do` | 决定不做了 |

#### 6b. 阻塞超时规则

`blocked` 状态不能无限期停留。**触发机制**：agent 在每次会话开始时检查当前 issue 的阻塞时长（通过 issue 文件中的 `blocked_since` 字段）。

- **阻塞超过 3 天**：agent 主动提醒用户，要求以下操作之一：
  - 升级：通知相关方 / 寻求帮助
  - 绕过：调整方案绕过阻塞点
  - 拆分：把不受阻塞的部分拆出来先做
- **阻塞超过 7 天**：必须重新评估该 issue 是否还值得继续，考虑降级为 `wont-do`

#### 6c. 流程回退机制

执行过程中如果发现需要调整，可以回退到之前的步骤：

| 触发情况 | 回退到 | 操作 |
|---|---|---|
| 执行中发现范围需要大幅调整 | Step 1 | 重新 grill，更新 issue |
| Review 发现验收标准定义有误 | Step 1 | 重新确认验收标准，更新 issue |
| 某个 slice 方案不可行 | Step 3 | 重新评估架构决策，可能需要新 ADR |
| 搜索发现已有现成方案，不需要自己写 | Step 4 | 重新评估外部参考，调整实现方案 |
| 阻塞无法解除且影响整体方案 | Step 1 | 重新评估范围和优先级 |
| 用户主动要求重新规划 | Step 1 | 从头 grill |

**回退决策权**：
- **小调整**（修改验收标准、调整 scope）：agent 可以直接做，事后通知用户
- **大调整**（改变架构方案、重新拆分 slices）：必须用户确认后再做

**产出**：`.scratch/issues/NNNN-xxx.md` 顶部 `role` 字段的正确更新。

### Step 7 — Review

**触发条件**：issue 进入 `ready-for-review` 状态。

**操作**：

1. **谁 review**：
   - 有明确 reviewer → 交给 reviewer
   - 独自开发 → agent 按验收标准逐项自检，给出 checklist
2. **Review 内容**（逐项检查）：
   - ✅ 验收标准是否全部达成？
   - ✅ 验证方式是否执行通过？（测试跑了吗？手动验证了吗？）
   - ✅ 有没有引入新的术语需要更新 CONTEXT.md？
   - ✅ 有没有需要记录的架构决策？（是否遗漏 ADR？）
   - ✅ 有没有引入技术债？（临时方案是否标注了后续清理计划？）
   - ✅ 有没有安全/权限变更？（新接口是否需要鉴权？数据是否脱敏？）
   - ✅ 有没有性能影响？（是否可能拖慢现有功能？）
   - ✅ 第三方库代码是否已对照文档验证？（或已标记可信度？）
3. **Review 结果**：
   - 通过 → 进入 `done`
   - 有反馈 → 切回 `in-progress`，记录反馈到 issue

**产出**：Review 结论，以及 issue 状态的正确更新。

---

### Step 8 — 完成后的清理

**触发条件**：issue 进入 `done` 状态。

**操作**：

1. 确认验收标准已达成（按 Step 1c 定义的验证方式执行）
2. 如果本次任务产生了新的术语，确认它们都已写入 `CONTEXT.md`
3. 如果本次任务做了重大架构决策，确认 ADR 已创建
4. 在 issue 底部加一行简短的"完成记录"（一句话，不需要长篇大论）

**产出**：一个干净关闭的 issue。

---

## 异常处理

| 情况 | 怎么做 |
|---|---|
| 用户说"先做了再说，不想写 issue" | 提醒：至少做 Step 1（grill），issue 和 CONTEXT.md 可以简化，但不要跳过 |
| 用户 grill 到一半就没耐心了 | 把已明确的部分先写进 issue，其余标记为待定；不要在不明确的情况下开始写代码 |
| 项目中没有 `.scratch/` 目录结构 | 按"目录结构"章节创建，并从附录中拷贝模板 |
| 用户不确定是否要写 ADR | 把三条门槛列给用户，让用户判断；用户拿不准时默认"先不写，观察一下" |
| 用户忘了切换 issue role | 提醒用户切换，或主动帮用户切换（后者更好） |
| 任务已在进行中，用户说"需求变了" | 回退到 Step 1 重新 grill，更新 issue（见 6c） |

## agent 自检清单

每次用户说"我要做 xxx"，agent 先在心里过一遍：

- [ ] 我有没有**先评估项目重要程度**，再决定 grill 深度？
- [ ] 我有没有**先 grill**，而不是直接写代码？
- [ ] grill 是不是**一次一个问题、按类型选交互方式**的？
- [ ] grill 完有没有**写 issue**？
- [ ] 验收标准里有没有明确**验证方式**（手动/自动化/混合）？
- [ ] 对话中有没有**术语歧义**？如果有，有没有**当场**更新 CONTEXT.md？
- [ ] 任务中有没有**重大架构决策**？如果有，有没有评估 ADR 三条门槛？（复用 Step 1a 的可逆性评估）
- [ ] 写代码前有没有做**外部参考评估**？第三方库 API 调用有没有对照文档？
- [ ] 如果没有搜索能力，有没有**标记代码可信度**并提醒用户验证？
- [ ] 任务是否**太大**（>2 天）？如果是，有没有提醒用户拆 slice？主 issue 是否设为 `parent`？
- [ ] issue 的 `role` 有没有**及时更新**？
- [ ] 有没有发生需要**回退**的情况？如果有，有没有正确处理？

## 附录 B — ADR 模板

> 模板文件位置：`.scratch/templates/ADR-FORMAT.md`

仅当以下**三个条件同时成立**时才创建一个 ADR：

1. **难逆转** — 将来改变主意的代价显著
2. **没有上下文会让人困惑** — 未来的读者会问"他们为什么这么做？"
3. **是真实的权衡** — 有多个合理备选方案，你因具体理由选了其中一个

缺一个就不要创建 ADR。

````markdown
# NNNN — 标题（一句话说明决策内容）

- 状态：Proposed / Accepted / Deprecated
- 日期：YYYY-MM-DD
- 替代：如果本 ADR 替代了之前的某个决策，在这里写明 `ADR-NNNN`

## 背景

描述当前面临的问题或需要决策的场景。说明"我们现在面临什么选择"。

## 决策

## 理由

- 理由 2：
- 理由 3：

## 备选方案与放弃原因

### 备选方案 A

描述。放弃原因：……。

### 备选方案 B

描述。放弃原因：……。

## 后果

### 好处

-

### 风险/代价

-

## 不做的事

明确列出本次决策**不**包含的范围，防止范围蔓延。
````

**状态流转**：

| 状态 | 含义 |
|---|---|
| **Proposed** | 已写好，在讨论中，尚未执行 |
| **Accepted** | 已决定，正在或已经按此执行 |
| **Deprecated** | 该决策被新的 ADR 取代，本文件保留作为历史记录 |

当一个 ADR 被取代时，**不要删除它**，把状态改为 Deprecated，并在顶部写上"被 ADR-NNNN 取代"。

## 附录 D — 可信度标记规范

当 agent 没有搜索能力，基于训练数据生成代码时，按以下格式标记可信度：

### 高可信度（训练数据中常见的标准用法）

```python
# ✅ 高可信度 — 标准库/常见用法，基于训练数据生成
# 建议对照 Python 官方文档确认
import json
data = json.loads(string)
```

### 中可信度（第三方库用法，可能有版本差异）

```python
# ⚠️ 中可信度 — 第三方库用法，基于训练数据生成
# 建议对照 requests 官方文档确认: https://docs.python-requests.org/
# 注意：API 参数可能因版本不同而有差异
import requests
response = requests.get(url, stream=True, timeout=30)
```

### 低可信度（不常见用法或复杂逻辑）

```python
# ❌ 低可信度 — 复杂逻辑，基于训练数据生成，未经外部验证
# 强烈建议：
# 1. 对照官方文档确认 API 用法
# 2. 编写测试覆盖边界情况
# 3. 在 staging 环境验证后再上线
```

## 附录 E — 技术栈术语增补约定

当项目使用特定技术栈时，相关的术语和约定写入 `tech/<技术栈>.md`，而非主 `CONTEXT.md`。

**好处**：
- 主 `CONTEXT.md` 保持精简，只放项目特有的术语
- 技术栈术语可以跨项目复用（如 `tech/python.md`）
- 切换技术栈时只需替换对应的 `tech/*.md`

**示例**：`tech/python.md`（见项目中的实际文件）
